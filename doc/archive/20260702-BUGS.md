# Prioritized Bug List

## [DONE] [P1] Raster serialization can panic on padded or non-zero-origin images

`GenericRasteriser.Serialise` pads odd image heights before iterating rows, but still calls `rasteriseLine` for padded rows. `bitmap.PixelBit` compares `x` and `y` against `img.Bounds().Dx()`/`Dy()` rather than absolute `Bounds().Max`, so padded rows or non-zero-origin images can call `img.At` out of bounds.

Proposed fix: make row/column bounds checks use absolute image bounds, or skip rasterizing padded rows and emit white bytes directly.

References: `raster.go`, `bitmap/image.go`.

## [DONE] [P1] Dry-run mode still requires Bluetooth

`bootstrap.Printer` enables the Bluetooth adapter before constructing the printer, even when `DRY_RUN=1` or `-dry` is set. This makes preview-only commands fail on machines without Bluetooth.

Proposed fix: skip `cfg.Adapter().Enable()` when `cfg.DryRun` is true.

Reference: `cmd/tp/internal/bootstrap/bootstrap.go`.

## [DONE] [P1] Composer options are ignored

`bitmap.NewComposer` accepts `ComposerOption` values but never applies them. As a result, compose options such as crop, dither function, and text dithering are silently ignored.

Proposed fix: loop over `opt` in `NewComposer` and apply each option to the newly created composer.

Reference: `bitmap/composer.go`.

## [DONE] [P1] `.image` command with no argument panics

`Document.cmdImage` rejects too many arguments but does not reject zero arguments before reading `args[0]`.

Proposed fix: require exactly one argument with `len(args) != 1`.

Reference: `bitmap/composer.go`.

## [DONE] [P2] IPP responses use a fixed request ID

IPP response helpers create messages with request ID `1` instead of echoing the incoming request ID. Clients may reject or mis-associate responses when multiple requests are in flight.

Proposed fix: pass `req.RequestID` into response creation and avoid the package-level `requestNum` constant for real responses.

References: `ippsrv/ipp_utils.go`, `ippsrv/ipp.go`.

## [DONE] [P2] Spool `RemoveJob` removes the job file twice

`RemoveJob` calls `removeJobLocked`, which already deletes the spool file, then attempts to delete the same file again and returns an error.

Proposed fix: remove the second `os.Remove` call, or split map cleanup and file deletion into separate helpers.

Reference: `ippsrv/spool.go`.

## [DONE] [P2] Spool lock is held while printing

`spool.AddJob` holds `s.mu` while triggering the job FSM. The FSM prints synchronously, so a long print blocks job listing, lookup, pruning, and other spool operations.

Proposed fix: register and write the job under lock, unlock, then start processing the job.

Reference: `ippsrv/spool.go`.

## [DONE] [P2] IPP error responses report success at the protocol level

`baseResponse` always constructs the message with `Code` set to `codeOK`, regardless of the `statusCode` argument. The passed-in status is only recorded as a non-standard `status-code` keyword attribute, so `scServerError` responses generated by the HTTP error path still carry `successful-ok` in the actual IPP status-code field. A conforming client reads the message `Code`, not this custom attribute, so it will treat failed requests (for example unsupported operations, printer lookup failures, or spool errors) as successful.

Proposed fix: change the response helper to accept concrete `goipp.Status` values and pass those into `goipp.NewResponse`, e.g. `goipp.StatusOk` for successful responses and `goipp.StatusErrorInternal`/`goipp.StatusErrorOperationNotSupported`/other specific IPP error statuses for failures. Remove the synthetic `"status-code"` operation attribute unless a spec-backed client compatibility reason requires it, and add tests that assert failed paths encode a non-success `Message.Code`.

References: `ippsrv/ipp_utils.go`, `ippsrv/http.go`, `ippsrv/ipp.go`.

## [DONE] [P2] Test suite depends on missing fixture files

`go test ./...` fails from a clean checkout because tests require `media/rasterised.png` and `media/harold.jpg`. One test also writes `../media/harold_out.png`.

Proposed fix: commit small deterministic fixtures, generate test images in memory, or skip external fixture tests with a clear message.

References: `raster_test.go`, `bitmap/dither_test.go`.

## [DONE] [P2] Print-Job response omits required job attributes

`handlePrintJob` returns a bare `baseResponse` with no `job-id`, `job-uri`, `job-state`, or `job-state-reasons`, all of which RFC 8011 section 4.2.1.2 requires in a Print-Job response. The job ID is generated server-side (`JobID(time.Now().Unix())`) and stored only in the spool, so a standard client that prints and then polls Get-Job-Attributes has no job ID to poll with and hangs or reports the job as failed even though it printed.

Proposed fix: after spooling, append the job's identifying attributes to the response job-attributes group, e.g. `resp.Job = job.attributes()` (or a minimal subset: job-id, job-uri, job-state, job-state-reasons).

Reference: `ippsrv/ipp.go` (`handlePrintJob`).

## [DONE] [P2] Get-Job-Attributes response is malformed

`handleGetJobAttributes` builds its response with `goipp.NewResponse` directly instead of `baseResponse`, then overwrites `resp.Operation` with `job.attributes()`. As a result the response lacks the mandatory `attributes-charset` and `attributes-natural-language` operation attributes (RFC 8011 section 4.1.4.1), and the job attributes are encoded in the operation-attributes group instead of the job-attributes group, so a conforming client reading the job group finds it empty.

Proposed fix: `resp = baseResponse(goipp.StatusOk, req.RequestID); resp.Job = job.attributes()`.

Reference: `ippsrv/ipp.go` (`handleGetJobAttributes`).

## [DONE] [P2] Get-Jobs merges all jobs into one attribute group

`handleGetJobs` appends every job's attributes to `resp.Operation`. RFC 8011 section 4.2.6.2 requires each job to be returned as its own job-attributes group; with two or more spooled jobs all `job-id`/`job-state` attributes concatenate into a single flat operation group with no delimiter between jobs, so clients cannot tell where one job ends and the next begins. goipp's `Message.Groups` field exists exactly for this case (its documentation cites the Get-Jobs response).

Proposed fix: build the response via `resp.Groups`, starting with the operation group from `baseResponse`, then one `goipp.Group{Tag: goipp.TagJobGroup, Attrs: job.attributes()}` per job.

References: `ippsrv/ipp.go` (`handleGetJobs`), `ippsrv/job.go`.

## [DONE] [P2] Client-caused failures are reported as server-error-internal-error

All handler errors funnel through a single mapping in `handlePrint` that encodes `goipp.StatusErrorInternal`. Client mistakes — missing `job-id`, unknown job (the `errJobNotFound` sentinel already exists), unknown printer, malformed `printer-uri` — should encode `client-error-bad-request` or `client-error-not-found`. Clients treat the 0x05xx class as a retryable server fault, so they retry or mark the printer failed instead of surfacing the user's error.

Proposed fix: let handlers express a status, e.g. a typed error carrying a `goipp.Status` mapped in `ServeIPP`, with `errors.Is(err, errJobNotFound)` → `client-error-not-found` and `printerFromRequest`/`extractValue` failures → `client-error-bad-request`; keep `server-error-internal-error` as the fallback.

References: `ippsrv/http.go`, `ippsrv/ipp.go`, `ippsrv/spool.go`.

## [DONE] [P3] Error-path encode failure is not logged; test assertions duplicated

In `handlePrint`, the error returned by `baseResponse(...).Encode(w)` on the ServeIPP-error path is discarded, while the success path four lines below logs encode failures. Separately, `ipp_test.go` copy-pastes the same RequestID/status assertion pair into four tests, comparing via `goipp.Code` (raw uint16 in failure output) instead of `goipp.Status` (which has a `String()` yielding RFC 8010 names).

Proposed fix: log the encode error for consistency; extract an `assertResponse(t, resp, wantID, wantStatus goipp.Status)` helper with `t.Helper()`.

References: `ippsrv/http.go`, `ippsrv/ipp_test.go`.

## [DONE] [P3] Critical battery log is unreachable

The low-battery branch checks `< 20` before `< 10`, so the critical branch can never run.

Proposed fix: check `< 10` before `< 20`.

Reference: `lx-d02.go`.
